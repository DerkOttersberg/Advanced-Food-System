================================================================================
                        DESIGN HIGHLIGHTS
================================================================================

WHY NO VANILLA POTION EFFECTS?
• Complete control over effect system
• Flexible (can create any effect imaginable)
• Clean UI (no vanilla potion effect icons)
• Easier combinations (fully trackable)

WHY CUSTOM BUFF SYSTEM?
• Apply effects via stat modifications
• Track in NBT (standard Minecraft save format)
• Client reads from synced NBT
• Render custom HUD instead of vanilla icons

WHY 3-SLOT HUD?
• Not overwhelming (information balance)
• Fits in corner (doesn't block vision)
• Auto-manages overflow (oldest first)
• Readable (large text, good contrast)

INTEGRATION APPROACH:
• Single event: Player eats food
• Two systems trigger: Vanilla hunger + custom buff
• They work together (not separate)
• Players understand instantly

================================================================================
                          IMPLEMENTATION HINTS
================================================================================

Based on the specifications, consider:

1. HEALTH SYSTEM
   • Modify player.getMaxHealth() on join
   • Listen to saturation changes
   • Add/remove bonus health dynamically

2. BUFF STORAGE
   • Use player.getPersistentData() for NBT
   • Store as ListTag of CompoundTags
   • One entry per buff (id, duration, magnitude, source)

3. FOOD EVENT
   • Hook LivingEntityUseItemEvent
   • Detect item type via registry name
   • Look up in food_buffs config
   • Add buff to player NBT

4. BUFF TICKER
   • Server tick event (every tick)
   • Decrement duration for each buff
   • Remove expired buffs
   • Apply effects based on buff type

5. HUD RENDERING
   • Client-side rendering
   • Read NBT from client player
   • Position and render slots
   • Format timer MM:SS
   • Update every frame

6. COMBINATIONS
   • Check every 5 ticks (4/second)
   • Scan player's active buff IDs
   • Match against combination rules
   • Apply bonus buff if matched

7. CONFIGURATION
   • Load JSON files on startup
   • Parse into Java objects
   • Validate values (clamp ranges, etc.)
   • Support reload command
